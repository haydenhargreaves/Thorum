Global Project Naming Conventions

structs

Structs will be named in CAPS

Examples:

ROW :: struct {}
EDITOR :: struct {}

Struct properties will be named in snake_case. This is to follow in C's footsteps.

Example:

ROW :: struct { len: int, render_len: int, }

Procedure Naming Conventions

Functionality will be broken into modules The module will determine specific rules for the naming convention.
But general rules are as follows: module.module_action_target

examples:

row.row_insert_char
row.row_remove_char
editor.editor_save_file
editor.editor_quit_editor
row.rows_insert_row NOTE: This one uses rows, for more info view the rows module naming convention

Module Specific Naming Conventions

row

This module contains functionality for the individual rows. In order to maintain code base readability and size, procedures in this module will be able to modify the row they are called with, as well as other rows. However, procedures that should modify every row will not be here.

To create a more readable code base, procedures that modify only a single row, the one provided in the parameters, the second module segment will remain as row.

Examples:

row.row_insert_char
row.row_remove_char
row.row_insert_substr
row.row_remove_substr

For procedures that modify more than one row, the second module segment will be altered to rows. Indicating that many rows will be modified. However, this does not mean many rows are passed into the procedures. This can be handled by the procedure itself, but the naming convention will apply regardless.

Examples:

row.rows_insert_newline
row.rows_remove_row

editor

This module contains functionality for the editor as a whole. It will serve as a sort of "wrapper" around most of the other functionality. The biggest difference in this module is that some procedures will not have a target and therefore procedures can omit the target in some cases. As an example, it would be silly to name a procedure editor.editor_init_editor. The final editor can be removed.

If can usually be inferred that when the target is omitted, that the target is the editor struct itself. However, this may not always be the case. It will be otherwise specified in the documentation if so.

Examples:

editor.editor_init
editor.editor_destory
editor.editor_save_file
editor.editor_open_file
